积分商城　前台需求规格（Share Treat）

版本 v1更新日 2026-06-25读者：QA / RD / UI

文件说明

这是 Share Treat 积分商城的会员端（前台）规格，供 QA / RD / UI 参考。

  • 核心改动：会员购买后，可在「订单详情」看到第三方派发的兑换信息（兑换码 / 兑换链接，或两者皆有），凭此去实体店核销；无密码字段。
  • 后台逻辑（选品 / 上架 / 审核 / 查看三方信息）见《后台需求规格（Share Treat）》。
QA 验收范围：本次改动仅针对积分商城「Share Treat」类型商品，验收请聚焦以下流程：购买 / 兑换条款同意 / 兑换信息展示 / 状态（待审核·已完成·已拒绝退分）/ 已搶光 / 余额不足。
特别验证：① 会员使用积分购买成功后，订单详情能正常显示兑换码或兑换链接（至少其一）；② 人工审核被拒后，积分历史出现「退还积分」记录。其他商品类型不在本次范围、行为维持不变。

改动说明

订单详情为既有页面，本次不新增页面，仅在原有「步骤条 + 订单信息」之下新增「兑换信息」区块。该区块展示第三方（BIZ TREATS）派发回传的数据，是会员实际「拿到货」的地方。

关键区分：「订单号」≠「兑换码」。「订单号」是识别这笔交易、用于查单 / 客服 / 对账；会员拿去实体店核销的是「兑换码 / 兑换链接」，不是订单号。两者在页面上分区块呈现，避免会员拿错。

订单信息区块（查单用 · 既有）

字段来源对应 API说明
订单号（自家）自家系统= 传给三方的 referenceNo原「兑换单号」改称「订单号」，与后台「商品兑换记录」列表的「订单号」用词一致；接三方前后都由自家生成，建议直接作为 referenceNo 传给三方，保证 1:1 对应
商品名称 / 价格自家 / 后台—沿用
三方单号（可选）三方bizReferenceNo默认不在会员端显示，放后台对账（后台已有「查看三方信息」）；仅当客服流程需会员报三方单号时，才以折叠 / 次要信息加入
单号命名约定：自家＝「订单号」（PR… ，referenceNo，与后台列表一致）；三方＝「三方单号 / 供应商单号」（260… ，bizReferenceNo，与后台用词一致）。避免两者都叫「单号」造成混淆。

兑换信息区块（换货用 · 新增）

以下字段全部由三方派发后回传，包含兑换码、兑换链接及有效期，内容以三方实际回传为准，我方不做预设或改写。

字段类型对应 API说明

功能逻辑

  • 如何兑换 / 兑换条款：兑换信息区块下方提供蓝字入口，点击弹出与商品详情页相同的「兑换说明」弹窗（含可兑换门市清单）。会员拿到兑换码后仍可随时查阅去哪里核销。
  • 不提供复制：兑换码 / 兑换链接不提供「复制」按钮，会员直接读取使用（连结型仍可「打开链接」）。
  • 打开链接：连结型 / 码＋链接型提供「打开兑换链接」；码＋链接型同屏展示兑换码。
  • 有效期：纯展示三方回传的 pinExpiredDate，时间格式化沿用既有逻辑；已过期时码 / 链接置灰并提示「已过期，请联系客服」。
  • 手动审核：审核通过后才会有兑换数据；此前区块为「审核中」占位，审核通过后刷新自动展示（呼应列表「数据同步存在延迟」）。
  • 审核拒绝（已拒绝）：后台人工审核拒绝后，订单状态转为「已拒绝」，系统全额退还本次扣减的积分至会员账户；该状态下不产生兑换数据，故不显示兑换码 / 链接 / 有效期，也不显示「如何兑换 / 兑换条款」。
  • 拒绝原因：会员端不显示拒绝原因；已拒绝详情页下方常驻「对订单有疑问？联系客服」，会员如需了解透过客服查询即可（后台「备注」为内部字段、非必填，沿用既有逻辑）。
  • 「联系客服」入口常驻。
