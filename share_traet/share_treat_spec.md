# 积分商城　完整需求规格（Share Treat）

**版本** v1　｜　**更新日** 2026-06-25　｜　**涉及端：** 前台 / 后台　｜　**读者：** QA / RD / UI

---

## 需求背景

积分商城原有奖励多为免费旋转券、免费投注券等，需绑定游戏 / 投注使用，使用场景较单一。本次接入第三方「Share Treat」，引入可直接核销的生活化品牌奖励，让积分兑换更有感、使用场景更丰富，提升会员对积分的兴趣与活跃度。


## 需求说明

**前台：**新增「Share Treat」分类，会员可用积分兑换商品；兑换后在订单详情查看兑换码 / 链接去核销。状态分待审核 / 已完成 / 已拒绝，拒绝时退还积分。

**后台：**支持上架 Share Treat 商品（数据来自第三方）、审核兑换（通过派发、拒绝退积分）、查看三方兑换信息（脱敏）。


## 文件说明

这是 Share Treat 积分商城的**会员端（前台）规格**，供 QA / RD / UI 参考。


- **核心改动：**会员购买后，可在「订单详情」看到第三方派发的**兑换信息**（兑换码 / 兑换链接，或两者皆有），凭此去实体店核销；无密码字段。
- 后台逻辑（选品 / 上架 / 审核 / 查看三方信息）见《后台需求规格（Share Treat）》。


> 💡 QA 验收范围：本次改动仅针对积分商城「Share Treat」类型商品，验收请聚焦以下流程：购买 / 兑换条款同意 / 兑换信息展示 / 状态（待审核·已完成·已拒绝退分）/ 已搶光 / 余额不足。特别验证：① 会员使用积分购买成功后，订单详情能正常显示兑换码或兑换链接（至少其一）；② 人工审核被拒后，积分历史出现「退还积分」记录。其他商品类型不在本次范围、行为维持不变。

## 第 1 页　商品列表

**路径：** `积分商城 > Share Treat 分类 · 商品以两列卡片并排展示；有折扣时左上角显示 ⚡ 标签与划线原价`



### 入口说明

会员在积分商城顶部点选「**Share Treat**」分类标签，即可看到后台已上架的商品列表。


| 项目 | 说明 |
| --- | --- |
| 分类标签 | 由后台控制是否显示「Share Treat」分类 |
| 商品来源 | 后台上架；上架 / 下架 / 排序均由后台控制 |
| 有折扣时 | 左上角显示 ⚡ 折扣标签（如 -20%）+ 划线原价 |
| 无折扣时 | 仅显示积分价格，无标签 |
| 点击商品卡 | 弹出商品详情弹窗（见第 2 页） |


### 商品卡字段（沿用）


| 字段 | 来源 | 说明 |
| --- | --- | --- |
| 商品图片 | 后台上传 | 由后台统一上传管理 |
| 商品名称 | 后台配置 | 由后台定义，如「McDonald's 中薯」「VP道具卡」 |
| 购买价格（积分） | 后台 | 会员需支付的积分数 |
| 积分折扣 / 限时 | 后台 | 既有逻辑，不变动；⚡ -XX% 标签 + 划线原价 |
| 库存 | 后台库存数值 | 前台展示后台库存数值；值为 0 时显示「已搶光」遮罩（见第 7 页） |


> ⚠️ 库存显示值：前台展示数据由后台库存数值决定，当值为 0 时，前台显示「已搶光」遮罩。


> 💡 QA 验证：后台需已上架至少一件 Share Treat 商品，才能在前台「Share Treat」分类看到商品。


### 文案长度限制


| 位置 | 限制 | 超出处理 |
| --- | --- | --- |
| 列表卡 · 商品名称 | **固定 1 行** | 超出宽度尾端以「…」省略（如：骰宝(ID: 44)免费旋转/投…） |
| 列表卡 · 库存 / 积分 | 各 1 行 | 正常显示，不受名称长度影响 |
| 详情弹窗 · 商品名称 | **最多 1 行** | 超出以「…」省略 |
| 详情弹窗 · 商品简介 | 不限行数 | 过长时弹窗内可滚动；无简介则隐藏该区域 |

商品名称在列表卡**只占 1 行**，名称太长时尾端以「…」省略（对应实际画面）。请勿因名称过长撑高卡片或换行，以保持两列卡片高度一致、整齐对齐；完整名称于**商品详情弹窗**呈现。


> 💡 QA 验证：上架一个超长名称商品，列表卡名称应压成 1 行 +「…」，且卡片高度与相邻卡片一致、不被撑高。


### 功能逻辑


- 库存为 0 时，商品卡显示「已搶光」遮罩（见第 7 页），且**商品卡不可点击**，无法进入商品详情。
- 商品下架时，商品卡不再展示于列表。

## 第 2 页　商品详情弹窗

**路径：** `商品列表 > 点击商品卡 · 弹窗含商品简介、「如何兑换 / 兑换条款」入口；积分充足可购买，不足时显示「余额不足」`



### 两种底部按钮状态


### 弹窗说明

点击商品卡弹出，沿用既有弹窗样式。本次新增：


- 标题栏显示分类名「**Share Treat**」（而非商品名）
- 商品名作为弹窗内容的副标题（左对齐，图片上方）
- 商品简介下方新增「**如何兑换 / 兑换条款 ›**」蓝字入口


### 弹窗字段来源


| 字段 | 来源 | 说明 |
| --- | --- | --- |
| 商品名称 | 后台配置 | 弹窗副标题，显示于图片上方左对齐；如「McDonald's 中薯」 |
| 商品图片 | 后台上传 / 三方 | 置中正方形缩图 |
| 商品简介 | 后台配置 | 图片下方简短描述文字；无配置时隐藏该区域 |
| 如何兑换 / 兑换条款 | 三方 Brand API `redeemInfoUrl` | 蓝字入口，点击弹出兑换说明二级弹窗 |
| 购买价格 | 后台 | 「购买」按钮内显示折后积分 |


### 新增：如何兑换 / 兑换条款


| 项目 | 规格 |
| --- | --- |
| 位置 | 商品简介下方、底部按钮上方 |
| 样式 | 蓝色文字 + 右箭头「›」，可点击 |
| 点击行为 | 弹出「兑换说明」二级弹窗（覆盖在商品详情之上） |


### 底部按钮两种状态


| 状态 | 触发条件 | 按钮文案 | 点击行为 | 关闭弹窗 |
| --- | --- | --- | --- | --- |
| 积分充足 | 会员积分 ≥ 商品折后价格 | 「购买 ⭐ XX」 | 跳转至**确认订单**（第 3 页） | 右上角「×」或点购买跳转 |
| 积分不足 | 会员积分 < 商品折后价格 | 「余额不足」 | 不可点击，无任何响应 | 仅右上角「×」可关闭 |


> ⚠️ 「余额不足」按钮外观与「购买」按钮一致（蓝底白字），仅文案不同且不可点击；会员须点右上角「×」关闭弹窗。


### 「兑换说明」弹窗规格


| 字段 | 来源 | 说明 |
| --- | --- | --- |
| 弹窗标题 | 固定 | 「如何兑换 / 兑换条款」 |
| 正文内容 | 三方提供的 `redeemInfoUrl` | 以 WebView 内嵌展示三方链接页面，**内容由三方定义，我方不管控** |
| 关闭 | 固定 | 右上角「×」返回商品详情 |


> ⚠️ 条款内容含使用限制、不可退款等说明，是客诉高发点，须购买前可见。条款文字样式由三方页面定义，我方不做干预。


### 功能逻辑


- 点「购买」进入**确认订单**（第 3 页）。
- 会员积分余额 < 商品折后价格时，「购买」按钮文案替换为「余额不足」且不可点，仅右上角「×」可关闭弹窗。
- 「兑换说明」弹窗内容拉取失败时，显示「加载失败，请稍后重试」+ 重试，不阻断商品浏览。

此弹窗以 **WebView** 内嵌方式加载三方提供的 `redeemInfoUrl`，页面内容由三方定义，我方仅负责加载展示，不做内容编辑。右图为示意，实际内容以三方页面为准。

## 第 3 页　确认订单

**路径：** `购买 > 确认订单弹窗`



### 说明

沿用既有确认订单结构（商品价格 / 折扣 / 总价 / 返回 / 付款），**新增「兑换条款同意」勾选**，置于「付款」按钮上方。


### 新增：兑换条款同意勾选


| 项目 | 规格 |
| --- | --- |
| 文案 | 「我已阅读并同意《如何兑换 / 兑换条款》」 |
| 《如何兑换 / 兑换条款》 | 蓝字可点，弹出与第 2 页相同的「兑换说明」弹窗 |
| 默认状态 | **不勾选**（不可预勾，须会员主动勾选） |
| 付款按钮 | 未勾选时**置灰不可点**；勾选后恢复可点 |


### 功能逻辑


- 未勾选同意：付款置灰；点击置灰按钮可轻提示「请先阅读并同意如何兑换 / 兑换条款」。
- 「付款」前再次校验库存与积分余额；不足时提示并阻止下单。
- 付款时同时校验三方库存数据与后台库存数据：任一方为 0 → 显示**购买失败弹窗**（第 4 页 · 状态 B），积分不扣除、不生成兑换记录。
- 库存充足 → 扣减会员积分并向后台 / 三方发起兑换（派发）。成功后进入**购买成功弹窗**（第 4 页 · 状态 A）。
- 建议服务端记录本次「同意条款」的时间戳与条款版本，便于日后查证。


> 💡 扣分与三方派发的时序、失败回滚（退积分）以后台 / 服务端为准；前台只负责呈现结果状态。

## 第 4 页　购买结果

**路径：** `付款 > 购买结果弹窗`



### 状态 A · 购买成功

移除「立即使用」按钮，**仅保留「查看订单」**，点击进入**兑换记录**（第 5 页）。


| 操作 | 行为 |
| --- | --- |
| 点击「查看订单」 | 进入兑换记录（第 5 页） |
| 点击弹窗外区域 | 弹窗关闭，停留在当前页面 |


> 💡 兑换信息（兑换码 / 兑换链接）统一在订单详情查看；购买成功后「查看订单」先进入兑换记录列表，再点入具体订单详情，入口收敛、体验一致。


### 状态 B · 购买失败（已搶光）

会员按下「付款」后，后端同时校验三方库存数据与后台库存数据，任一方为 **0** 时，显示此失败弹窗。


| 项目 | 规格 |
| --- | --- |
| 标题 | 已搶光（共用第 7 页「已搶光」翻译 key） |
| 说明文字 | 很抱歉，商品库存已不足，购买未成功 |
| 按钮 | 「返回」 |
| 积分 | **不扣除**（校验在扣分前执行） |
| 兑换记录 | **不生成** |


| 操作 | 行为 |
| --- | --- |
| 点击「返回」 | 返回商品列表（第 1 页）；该商品显示「已搶光」遮罩、库存 0 |
| 点击弹窗外区域 | 同上，返回商品列表 |


> 💡 积分校验顺序：先验库存 → 再扣积分，避免扣分后无货的回滚复杂度。

## 第 5 页　兑换记录

**路径：** `积分商城 > 兑换记录；或 购买成功 > 查看订单`



### 改动点


- **状态标签**：沿用系统既有三种状态（已完成 / 待审核 / 已拒绝）。自动审核 → 购买后直接显示「已完成」；人工审核 → 先显示「待审核」，审核通过后更新为「已完成」，审核拒绝则更新为「已拒绝」（系统全额退还本次扣减的积分）。
- 其余结构（缩略图、时间、商品名、消耗积分、点击进详情）沿用既有逻辑。


### 字段规格


| 字段 | 说明 |
| --- | --- |
| 缩略图 | 商品图，缺图显示占位图 |
| 下单时间 | `YYYY-MM-DD HH:mm:ss` |
| 商品名称 | 如「McDonald's 中薯」 |
| 消耗积分 | 本笔扣减的积分 |
| 状态 | 待审核 / 已完成 / 已拒绝 |


### 功能逻辑


- 顶部下拉可按状态筛选（全部 / 待审核 / 已完成 / 已拒绝）。
- 三种状态（待审核 / 已完成 / 已拒绝）的记录均可点击，进入**订单详情**（第 6 页）。
- **已拒绝**：人工审核被拒的订单，列表显示灰色「已拒绝」标签；该笔扣减的积分已由系统全额退还至会员账户。
- 保留「数据同步存在延迟」提示文案。


> 💡 前后台状态对照：前台「已完成」＝后台「审核通过」；前台「待审核」＝后台「待审核」；前台「已拒绝」＝后台「审核未通过」。三方派发、扣分与退分的时机以后台 / 服务端为准。

## 第 6 页　订单详情（既有页，新增「兑换信息」区块）

**路径：** `兑换记录 > 订单详情　此页本已存在，本次仅在其内新增兑换信息`



### 三种状态画面对比


### 改动说明

**订单详情为既有页面，本次不新增页面**，仅在原有「步骤条 + 订单信息」之下**新增「兑换信息」区块**。该区块展示第三方（BIZ TREATS）派发回传的数据，是会员实际「拿到货」的地方。


> ⚠️ 关键区分：「订单号」≠「兑换码」。「订单号」是识别这笔交易、用于查单 / 客服 / 对账；会员拿去实体店核销的是「兑换码 / 兑换链接」，不是订单号。两者在页面上分区块呈现，避免会员拿错。


### 订单信息区块（查单用 · 既有）


| 字段 | 来源 | 对应 API | 说明 |
| --- | --- | --- | --- |
| 订单号（自家） | **自家系统** | = 传给三方的 `referenceNo` | 原「兑换单号」改称「订单号」，**与后台「商品兑换记录」列表的「订单号」用词一致**；接三方前后都由自家生成，建议直接作为 referenceNo 传给三方，保证 1:1 对应 |
| 商品名称 / 价格 | 自家 / 后台 | — | 沿用 |
| 三方单号（可选） | 三方 | `bizReferenceNo` | 默认**不在会员端显示**，放后台对账（后台已有「查看三方信息」）；仅当客服流程需会员报三方单号时，才以折叠 / 次要信息加入 |


> 💡 单号命名约定：自家＝「订单号」（PR… ，referenceNo，与后台列表一致）；三方＝「三方单号 / 供应商单号」（260… ，bizReferenceNo，与后台用词一致）。避免两者都叫「单号」造成混淆。


### 兑换信息区块（换货用 · 新增）

以下字段**全部由三方派发后回传**，包含兑换码、兑换链接及有效期，内容以三方实际回传为准，我方不做预设或改写。


| 字段 | 类型 | 对应 API | 说明 |
| --- | --- | --- | --- |
| 兑换码 | 文本（纯展示） | `pin` | **码型：会员出示此码给店员核销**；与链接并存时，会员可同时看到兑换码与兑换链接 |
| 兑换链接 | 文本（纯展示） | `bizRedeemUrl` | **连结型：会员打开此链接兑换**，导向供应商兑换页；提供「打开链接」按钮 |
| 有效期至 | 文本（纯展示） | `pinExpiredDate` | 三方派发时回传的到期日，**纯展示**，不由我方计算；时间格式化沿用既有逻辑；到期后无需置灰 |


### 按商品类型（pinType）显示规则


| 类型 | pinType | 显示内容 |
| --- | --- | --- |
| PIN 码型 | `pin` | 仅显示**兑换码**（+ 有效期） |

## 页面 1　新增商品

**路径：** `后台 > 积分商城 > 商品管理 > 新增商品`



### 改动说明

新增商品类别 **Share Treat** 及商品类型 **Share Treat**。选中商品类型 Share Treat 后，在下方显示一组「三级联动 + 商品详情」字段，供管理员选定要上架兑换的商品。其余字段（商品图片、购买价格、积分折扣比例、排序、单个会员限购数、库存、是否限时抢购、审核模式等）维持既有逻辑不变。


### 改动点


- 新增商品类别 **Share Treat** 及商品类型 **Share Treat**。
- 选中 Share Treat 后，新增字段：选择类别、选择厂商、选择商品、商品详情、商品简介。
- 选中其他商品类型时，上述新增字段不显示。


### 字段规格


| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 选择类别 Category | 下拉 | 是 | Share Treat 商品类别，数据来自第三方 API |
| 选择厂商 Brand | 下拉 | 是 | 依所选类别联动带出 |

| 库存 | 数字 | 是 | 管理员手动输入，输入值不可超过第三方 API 回传库存；输入框后方显示灰色辅助文字「剩余库存：xxx 件」，数值来自第三方 API 实时回传，供管理员填写时参考。若第三方 API 回传库存为 **0**，辅助文字显示「剩余库存：0 件」，「库存」栏位自动置灰、不可填写；若第三方 API **无回传值**，辅助文字不显示，「库存」栏位同步置灰、不可填写 |
| 商品简介 | 文本 | 是 | 沿用既有字段 |


### 功能逻辑


- 联动顺序：类别 → 厂商 → 商品 → 详情。
- 上层变更时，下层选项需清空并重新加载。
- 类别 / 厂商 / 商品三级数据均来自第三方提供的 API（接口文件）。
- 库存辅助文字：选定商品后，输入框后方自动显示灰色文字「剩余库存：xxx 件」，数值实时来自第三方 API 回传，供管理员填写时参考。
- 库存上限验证：管理员输入值不可超过第三方 API 回传库存；超出时栏位下方即时显示红字「库存不可超过三方库存上限 {N}」，「确定」按钮同步 disabled，须修正后方可提交。
- 若第三方 API 回传库存为 **0**，辅助文字显示「剩余库存：0 件」，「库存」输入栏自动置灰、不可填写。
- 若第三方 API **无回传值**，辅助文字不显示，「库存」输入栏自动置灰、不可填写。


### 异常处理（第三方 API 连接失败）

选择 Share Treat 后，若三级下拉的数据来源（第三方 API）连接失败或无响应，需弹出错误提示，并将受影响的下拉置为不可选、「确定」按钮禁止提交。


- 错误提示（顶部 Toast）：**「资料载入失败，请检查网络或稍后重试」**
- 连接失败期间「确定」按钮禁止提交，避免存入空商品。

## 页面 3　商品兑换记录

**路径：** `后台 > 积分商城 > 商品兑换记录`



### 改动说明

针对 Share Treat 类型商品，在类型筛选下拉新增选项，并在「操作」栏提供查看第三方（供应商）兑换信息的入口。弹窗样式参考既有「充值信息」弹窗。


### 改动点


- **类型筛选下拉（最右侧）**：新增选项 **Share Treat**，可按此类型筛选。
- **操作栏**：Share Treat 类型订单显示「**查看三方信息**」（对应原话费类的「查看充值信息」）。
- **审核控制**：手动审核时，操作栏**先显示「审核」按钮**，完成审核后才显示「查看三方信息」；自动审核直接显示「查看三方信息」。
- **点击「查看三方信息」**：弹出「三方信息」窗口，显示三方订单号、兑换码 / 兑换链接。


### 审核动作（手动审核：通过 / 拒绝）

手动审核的订单，操作栏先显示「**审核**」按钮。点击后进入审核处理，管理员可选择**通过**或**拒绝**，并可填写「备注」（**非必填，沿用既有逻辑**）。审核结果与后续行为如下：


| 审核结果 | 订单状态 | 积分 | 三方派发 | 操作栏 | 前台对应 |
| --- | --- | --- | --- | --- | --- |
| **通过** | 审核通过 | 维持已扣减 | 触发三方派发兑换数据 | 改显「查看三方信息」 | 已完成 |
| **拒绝** | **审核未通过** | **全额退还本次扣减积分** | 不派发、无兑换数据 | 不显示「查看三方信息」 | 已拒绝 |


### 订单状态与操作人


| 订单状态 | 含义 | 操作人显示 |
| --- | --- | --- |
| 待审核 | 手动审核、尚未处理 | —（待处理） |
| 审核通过 | 自动审核通过，或管理员审核通过 | **自动＝**`SYSTEM`；**人工＝**管理员账号（如 `vivianadmin`） |
| 审核未通过 | 管理员审核拒绝 | 执行拒绝的管理员账号 |


### 前后台状态对照


| 后台（订单状态） | 前台（兑换记录 / 订单详情） |
| --- | --- |
| 待审核 | 待审核 |
| 审核通过 | 已完成 |
| **审核未通过** | **已拒绝** |


### 弹窗字段规格


| 字段 | 类型 | 说明 | 脱敏规则 |
| --- | --- | --- | --- |
| 三方订单号 | 文本 + 复制 | 供应商回传的订单号，供对账追溯；点击复制按钮可复制**完整订单号**（与既有复制逻辑一致） | 显示时仅呈现**后五码**，前段以 `*` 遮蔽；复制内容为完整值 |
| 兑换码 | 文本（不可复制） | 可兑换的 PIN / 券码（码型商品） | 仅显示**后四码**，前段以 `*` 遮蔽；移除复制按钮 |
| 兑换链接 | 链接（不可复制） | 导向供应商兑换页（链接型商品） | 仅显示**后四码**，前段以 `*` 遮蔽；移除复制按钮 |


### 功能逻辑


- 兑换码 / 兑换链接依第三方 API 返回的数据显示，二者可能其一、也可能并存（取决于该商品的 API 数据）。
- 「查看三方信息」仅 Share Treat 类型显示；其他类型维持原入口。
- **脱敏显示**：三方信息弹窗中的敏感字段均需模糊处理——三方订单号仅显示后五码，兑换码 / 兑换链接仅显示后四码；兑换码与兑换链接同时移除复制按钮，防止明文外泄。
- **三方订单号复制**：点击复制按钮后，复制内容为**完整三方订单号**（非脱敏后的显示值），与既有复制逻辑一致。
- **审核拒绝退积分**：手动审核拒绝时，系统全额退还会员本次扣减的积分；订单状态记为「审核未通过」，操作人记为执行拒绝的管理员账号，前台同步显示「已拒绝」。
