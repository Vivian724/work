# 积分商城　完整需求规格（Share Treat · 前台＋后台）

**版本** v2　｜　**更新日** 2026-07-01　｜　**涉及端：** 前台 / 后台　｜　**读者：** QA / RD / UI

> 本版对应「新流程」：会员购买时**自行填写手机号码与邮箱**，由 ShareTreats 直接把兑换信息派发给会员；我方**不再持有、不回传兑换码 / 兑换链接**，订单详情只展示**三方状态**。

---

## 一、需求背景

积分商城原有奖励多为免费旋转券、免费投注券等，需绑定游戏 / 投注使用，场景单一。本次接入第三方「Share Treat」，引入可直接核销的生活化品牌奖励，提升会员对积分的兴趣与活跃度。

## 二、需求说明

- **前台：** 新增「Share Treat」分类，会员可用积分兑换商品；购买时填写手机号码与邮箱，三方凭此自行派发兑换信息；订单详情可查看三方状态（前台仅 成功 / 失败 / —；后台「处理中」前台以「—」呈现）。状态分待审核 / 已完成 / 已拒绝，**人工拒绝或兑换失败时全额退还积分**。
- **后台：** 支持上架 Share Treat 商品（数据来自第三方）、审核兑换（通过派发、拒绝退积分）、商品兑换记录新增「三方状态」列与筛选、查看订单信息（含重新发送）。

> **QA 验收范围：** 仅针对积分商城「Share Treat」类型商品。聚焦：购买 / 手机号码与邮箱填写验证 / 兑换条款同意 / 三方状态展示 / 状态（待审核·已完成·已拒绝退分）/ 兑换失败自动退分 / 已搶光 / 余额不足。其他商品类型不在本次范围、行为维持不变。

---

## 三、流程总览

```
① 会员      填手机号码 + 邮箱 → 付款
                ↓
② 我方后台   审核通过（自动 / 人工）→ 送给 ShareTreats
                ↓
③ ShareTreats 把兑换码发到会员手机 / 邮箱
                ↓
④ callback   三方回传结果
             · 成功 →（可在后台「重新发送」再发一次）
             · 失败 → ＝兑换失败，系统自动退还积分
```

**兑换失败（callback 失败 → 自动退分）的原因：**
- ❶ 送不出去（网络逾时 / 三方异常 / 缺货 / 下架）
- ❷ 送不达（无效号码、退信）

**注意 — 填错到「有效但错误」的地址：** 券会照发到该错误号码 / 邮箱，三方回传**成功**，**不退款**（见确认订单页「信息错误恕不退款」警语）。这不算失败。

## 四、核心状态模型（前后台对照）

订单详情有**两条状态**：
- **订单状态**：待审核 / 已完成 / 已拒绝（前台）＝ 待审核 / 审核通过 / 审核未通过（后台），由我方审核决定。
- **三方状态**：后台 处理中 / 成功 / 失败 / —（由 ShareTreats 回传 callback 决定）；**前台仅显示 成功 / 失败 / —，「处理中」前台以「—」呈现**。

**前台画面状态由「三方状态值」映射而来**（三方状态是后台取 ShareTreats 回传写入的真值）：

| 后台三方状态（真值） | 后台情形 | 前台画面 | 前台三方状态栏 | 积分 | 备注 |
| --- | --- | --- | --- | --- | --- |
| **—** | 人工审核中 | 待审核 | — | 已扣减 | 尚未发送给 ShareTreats |
| **处理中** | 审核通过 → 已发、等待 callback | **待审核** | **—**（处理中不给会员看） | 已扣减 | 画面仍停「待审核」，callback 成功才转「已完成」 |
| **成功** | 审核通过 → callback 成功（兑换成功） | 已完成 | 成功 | 已扣减 | 兑换信息已发至会员手机 / 邮箱 |
| **失败** | 审核通过 → callback 失败（兑换失败） | 已拒绝 | 失败 | **全额退还** | 三方已回滚；**后台订单状态维持「审核通过」**，前台画面显示「已拒绝」，会员可重新购买 |
| **—** | 人工审核**拒绝** | 已拒绝 | — | **全额退还** | 从未发送；已退分 |

> **前台三方状态栏只有「—／成功／失败」**：后台的「处理中」是内部中间态，前台不显示、一律以「—」呈现（会员看到待审核画面 + 三方状态「—」）。
> **两轴独立：** 后台订单状态只反映我方审核结果（待审核 / 审核通过 / 审核未通过）；三方状态由 ShareTreats 独立决定。
> **兑换失败＝审核通过＋三方状态失败**：订单状态维持「审核通过」、三方状态「失败」、系统自动退分、前台画面「已拒绝」。
> 前台「已拒绝」有两种来源：① 人工拒绝（后台审核未通过、三方状态＝—）；② 兑换失败（后台审核通过、三方状态＝失败）。两者积分均全额退还。

## 五、前台规格

### 页 1 · 商品列表（积分商城 > Share Treat 分类）
沿用图片 / 价格 / 折扣 / 库存；点击商品进入商品详情。库存为 0 时商品卡叠加「已搶光」遮罩且不可点击。

### 页 2 · 商品详情（商品列表 > 点击商品卡）
新增「如何兑换 / 兑换条款」入口 + 二级「兑换说明」弹窗（内容来自三方 Brand API）；积分不足时底部按钮显示「余额不足」，仅「×」可关闭。

### 页 3 · 确认订单（购买 > 确认订单弹窗）
沿用既有确认订单结构，**新增手机号码、邮箱输入框与「兑换条款同意」勾选**，均置于「付款」按钮上方。三方凭会员填写的手机号码 / 邮箱自行派发兑换信息。

**新增：手机号码 / 邮箱输入框**

| 项目 | 规格 |
| --- | --- |
| ＊手机号码 | 必填；placeholder「请输入」；标签前方显示红色星号「*」 |
| ＊邮箱 | 必填；placeholder「请输入」；标签前方显示红色星号「*」 |
| 空格限制 | 两个输入框均**不允许输入空格**；前端实时静默去除任何空格字符（含全角空格） |
| 手机号码格式验证 | 仅限菲律宾本地号码，**最多 11 码**；超出第 11 位自动截断；付款前验证长度须恰好 11 码，不符时于**输入框下方**显示「格式错误」 |
| 邮箱格式验证 | 须为正确邮箱格式（如 `xxx@gmail.com`）；不符时于**输入框下方**显示「格式错误」 |
| 提示文案 | 输入框下方红色文字：「购买前请确保您的手机号码及邮箱正确无误，信息错误恕不退款。」 |

**「付款」按钮启用条件**（三项同时满足才可点，任一不满足置灰）：
1. 手机号码已填写且格式正确（11 码）
2. 邮箱已填写且格式正确
3. 「我已阅读并同意《如何兑换 / 兑换条款》」已勾选

**功能逻辑**
- 付款前再次校验库存与积分余额；不足时提示并阻止下单。
- 付款时同时校验三方库存与后台库存：任一为 0 → 购买失败弹窗（页 4 · 状态 B），积分不扣、不生成记录。
- 库存充足 → **扣积分、生成兑换记录**，进入购买成功弹窗（页 4 · 状态 A）；后续是否立即派发依**该商品后台审核配置**：
  - **自动审核**：系统直接把手机 / 邮箱传给三方派发（订单状态＝审核通过）；三方 callback 成功后前台显「已完成」、失败则自动退分、前台「已拒绝」（回传前为处理中→前台仍显「待审核」）。
  - **人工审核**：订单先进「待审核」，管理员通过后才派发（之后同自动审核的 callback 流程）；拒绝则不派发、全额退分、前台「已拒绝」。

### 页 4 · 购买结果（付款 > 购买结果弹窗）
- 状态 A · 购买成功：移除「立即使用」，仅保留「查看订单」，点击进入**兑换记录**页面。
- 状态 B · 购买失败（已搶光）：三方或后台库存任一为 0 时显示。

### 页 5 · 兑换记录（积分商城 > 兑换记录）
状态沿用系统既有三种（已完成 / 待审核 / 已拒绝）。审核通过后由三方 callback 决定：成功→「已完成」、失败→「已拒绝」（自动退分）；**callback 回传前（处理中）仍显「待审核」**。人工审核先「待审核」，拒绝则「已拒绝」（全额退还）。

- **已拒绝（两种来源）**：① 人工审核被拒；② 审核通过但三方兑换失败。两者均显示灰色「已拒绝」标签，扣减积分皆已全额退还。

### 页 6 · 订单详情（兑换记录 > 订单详情，既有页，非新增页）
本页新增：手机号码、邮箱（购买时填写值，只读）、三方状态。**前台画面状态由三方状态映射**（见第四节表格）。

**四种状态画面对比**（示意图顺序）：待审核 → 已拒绝·人工 → 已拒绝·兑换失败 → 已完成。已拒绝有两种来源，画面相同但三方状态不同（— / 失败）；后台「处理中」时前台画面仍显示「待审核」、三方状态栏显示「—」（处理中不给会员看）。

**功能逻辑**
- 手机号码 / 邮箱只读，不提供修改入口。
- 前台画面由三方状态映射：处理中→待审核、成功→已完成、失败→已拒绝、—（审核中/人工拒绝）→待审核/已拒绝。**前台三方状态栏只有 —／成功／失败，处理中一律以「—」呈现（不给会员看）。**
- 三方状态随 callback 自动更新，无需会员操作。
- **三方状态失败＝兑换失败**：系统自动全额退还积分，**前台画面显示「已拒绝」**（后台订单状态仍为「审核通过」），会员可用退回积分重新购买；积分历史出现「退还积分」记录，详情页常驻「联系客服」。

### 页 7 · 商品列表 · 已搶光
库存为 0 时商品卡叠加半透明遮罩显示「已搶光」，**商品卡不可点击**。

### 页 8 · 积分历史（积分商城 > 积分历史）
Share Treat 购买产生两类条目：

| 条目 | 触发时机 | 积分变动 | 系统文案 |
| --- | --- | --- | --- |
| 扣分 | 会员付款成功（含自动 & 人工审核商品） | 红色负数，如 −16 | 「Redeem {商品名}」 |
| 退分 | **两种情形**全额退还：① 人工审核拒绝；② 兑换失败（callback 失败自动退分） | 绿色正数，与扣分同额，如 +16 | 「Points refunded after product review rejection」 |

> **退分的两个来源**：（1）人工审核被拒（仅人工审核商品）；（2）兑换失败自动退分（自动 / 人工审核商品都可能发生）。因此自动审核商品也可能因兑换失败产生退分记录。

## 六、后台规格

### 页 1 · 新增商品（后台 > 积分商城 > 商品管理 > 新增商品）
新增商品类别及商品类型 **Share Treat**。选中后显示「类别 / 厂商 / 商品」三级联动 + 商品详情。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 选择类别 Category | 下拉 | 是 | 数据来自第三方 API |
| 选择厂商 Brand | 下拉 | 是 | 依所选类别联动 |
| 选择商品 Goods | 下拉 | 是 | 依所选厂商联动 |
| 商品详情 | 只读 | — | 选定后自动带出，不可编辑 |
| 库存 | 数字 | 是 | 手动输入，不可超过三方 API 回传库存；辅助文字「剩余库存：xxx 件」；API 回传 0 或无回传值时库存栏置灰不可填 |
| 商品简介 | 文本 | 是 | 沿用既有字段 |

**功能逻辑**：联动顺序类别→厂商→商品→详情（上层变更清空下层）；库存上限验证（超出置灰不可提交）；API 连接失败时顶部 Toast 提示、禁止提交。

### 页 2 · 用户积分记录（后台 > 积分商城 > 用户积分记录）
Share Treat 订单「原因」显示商品类型 `Share Treat`；「备注」显示商品名称。

### 页 3 · 商品兑换记录（后台 > 积分商城 > 商品兑换记录）

**改动点**
1. 类型筛选下拉新增选项 **Share Treat**。
2. **列表「充值状态」列改名沿用为「三方状态」列**：显示 ShareTreats 回传结果——成功（绿）/ 失败（红）/ 处理中（灰）/ —（未派发）。
3. **搜索栏新增「三方状态」筛选下拉**：选项 成功 / 失败 / 处理中。
4. 操作栏显示「查看订单信息」（弹出订单信息窗口）。

**操作栏按状态区分**
- **仅「待审核」＝「审核」按钮；其余状态（审核通过 / 审核未通过）一律＝「查看订单信息」**
- 人工拒绝的订单三方订单号＝无、三方状态＝—，但会员手机 / 邮箱仍可查看

**订单状态总览**（后台订单状态＝我方审核结果；三方状态独立）

| 后台订单状态 | 触发条件 | 可能的三方状态 | 前台画面 | 积分 | 操作栏 | 操作人 |
| --- | --- | --- | --- | --- | --- | --- |
| 待审核 | 手动审核中 | —（未派发） | 待审核 | 已扣减 | 「审核」按钮 | — |
| 审核通过 | 自动 / 人工通过 → 已派发 | 处理中 | 待审核 | 已扣减 | 「查看订单信息」 | 自动＝SYSTEM；人工＝管理员账号 |
| 审核通过 | 同上 | 成功 | 已完成 | 已扣减 | 「查看订单信息」 | 同上 |
| 审核通过 | 同上，callback 失败（兑换失败） | 失败 | 已拒绝 | **全额退还** | 「查看订单信息」 | 同上 |
| 审核未通过 | 管理员人工拒绝 | —（从未派发） | 已拒绝 | 全额退还 | 「查看订单信息」 | 执行拒绝的管理员账号 |

> **积分退还**：只要退分即为**全额退还**（人工拒绝、兑换失败皆同），不分自动 / 手动。
> **操作栏**：仅「待审核」显示「审核」按钮；其余状态一律「查看订单信息」——人工拒绝虽无三方订单号 / 状态，但会员手机、邮箱仍可查看。

**订单信息弹窗字段**

| 字段 | 说明 | 操作 |
| --- | --- | --- |
| 三方订单号 | ShareTreats 回传订单号，供对账追溯 | — |
| 三方状态 | ShareTreats 回传结果（成功 / 失败 / 处理中 / —） | — |
| 手机号码 | 会员购买时填写，完整显示 | 「重新发送」——**仅三方状态＝成功时显示**；失败 / 处理中 / — 时不显示此按钮 |
| 邮箱 | 会员购买时填写，完整显示 | 「重新发送」——**仅三方状态＝成功时显示**；失败 / 处理中 / — 时不显示此按钮 |

**功能逻辑**
- **三方状态列取值时机**：订单**审核通过后（自动 / 人工皆同）**，系统向 ShareTreats 发起派发并取回传状态填入「三方状态」列——已送出待回传＝处理中，callback 回来后转成功 / 失败；未审核通过（待审核 / 人工拒绝）＝—。
- **三方状态筛选**：搜索栏下拉可按 成功 / 失败 / 处理中 过滤。
- **重新发送（仅限三方状态＝成功）**：仅当三方状态为「成功」时，手机号码 / 邮箱后方才显示「重新发送」按钮，用途是把已成功的兑换信息**再发一次**给会员（会员遗失简讯 / 邮件时）。点击后系统调用 ShareTreats Resend 接口请三方重派（发码方仍为 ShareTreats）。三方状态**非成功**（处理中、失败等）时弹窗内**不显示此按钮**；成功但达每日上限时按钮置灰。（待审核订单操作栏为「审核」、不进入本弹窗。）
- **兑换失败自动退分**：callback 失败＝兑换失败、三方已回滚，系统自动全额退还积分（无需管理员操作）。此时**后台订单状态维持「审核通过」**、三方状态「失败」、**前台画面显示「已拒绝」**；会员可用退回积分重新购买。故失败不提供重新发送，也无「填错改址补发」需求。

> ⚠ **待开发确认（对接 ShareTreats API）：** ShareTreats 提供两支重发接口——SMS `POST /api/v3/sms/resend/{referenceNo}`、Email `POST /api/v3/email/resend`。**每日重发次数上限（SMS 文件写 2 次 / 日）与计算范围（整笔订单共用 or 手机、邮箱各自计）、剩余次数是否以三方回传值为准**，请开发对照三方文件确认后回填本规格。凭 `reference_no`（＝三方订单号）指定重发笔。

## 附：与旧流程的差异（重点）

| 项目 | 旧流程 | 新流程（本版） |
| --- | --- | --- |
| 兑换信息来源 | 我方接收三方兑换码 / 链接并展示给会员 | 会员填手机 / 邮箱，**ShareTreats 直接派发给会员** |
| 订单详情展示 | 兑换码 / 兑换链接 / 有效期 | 手机号码 / 邮箱 / **三方状态** |
| 失败处理 | —— | callback 失败＝兑换失败，**自动全额退分**（后台维持审核通过、前台显示已拒绝） |
| 重新发送 | —— | 仅三方状态成功时可用（重发已成功的信息） |
