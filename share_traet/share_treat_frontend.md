# 积分商城　前台需求规格（Share Treat）

**版本** v1　｜　**更新日** 2026-06-25　｜　**读者：** QA / RD / UI

---

## 文件说明

这是 Share Treat 积分商城的**会员端（前台）规格**，供 QA / RD / UI 参考。

- **核心改动：** 会员购买后，可在「订单详情」看到第三方派发的**兑换信息**（兑换码 / 兑换链接，或两者皆有），凭此去实体店核销；无密码字段。
- 后台逻辑（选品 / 上架 / 审核 / 查看三方信息）见《后台需求规格（Share Treat）》。

> **QA 验收范围：** 本次改动**仅针对积分商城「Share Treat」类型商品**，验收请聚焦以下流程：购买 / 兑换条款同意 / 兑换信息展示 / 状态（待审核·已完成·已拒绝退分）/ 已搶光 / 余额不足。  
> 特别验证：① 会员使用积分购买成功后，订单详情能正常显示兑换码或兑换链接（至少其一）；② 人工审核被拒后，积分历史出现「退还积分」记录。**其他商品类型不在本次范围、行为维持不变。**

---

## 第 1 页　商品列表

**路径：** `积分商城 > Share Treat 分类`  
商品以两列卡片并排展示；有折扣时左上角显示 ⚡ 标签与划线原价。

### 入口说明

会员在积分商城顶部点选「**Share Treat**」分类标签，即可看到后台已上架的商品列表。

| 项目 | 说明 |
|------|------|
| 分类标签 | 由后台控制是否显示「Share Treat」分类 |
| 商品来源 | 后台上架；上架 / 下架 / 排序均由后台控制 |
| 有折扣时 | 左上角显示 ⚡ 折扣标签（如 -20%）+ 划线原价 |
| 无折扣时 | 仅显示积分价格，无标签 |
| 点击商品卡 | 弹出商品详情弹窗（见第 2 页） |

### 商品卡字段（沿用）

| 字段 | 来源 | 说明 |
|------|------|------|
| 商品图片 | 后台上传 | 由后台统一上传管理 |
| 商品名称 | 后台配置 | 由后台定义，如「McDonald's 中薯」「VP道具卡」 |
| 购买价格（积分） | 后台 | 会员需支付的积分数 |
| 积分折扣 / 限时 | 后台 | 既有逻辑，不变动；⚡ -XX% 标签 + 划线原价 |
| 库存 | 后端计算后回传 | 前台展示后端回传值；值为 0 时显示「已搶光」遮罩（见第 7 页） |

> **库存显示值（归属后台规格书）：** 前台展示的库存数字由后端统一计算后回传，前台不做任何运算。取值原则：以「三方实时库存」与「后台配置库存上限」两者中较小的值为准。回传值为 0 时，前台显示「已搶光」遮罩。详细取值逻辑请定义于后台规格书，本文件不重复说明。

> **QA 验证：** 后台需已上架至少一件 Share Treat 商品，才能在前台「Share Treat」分类看到商品。

### 文案长度限制

| 位置 | 限制 | 超出处理 |
|------|------|----------|
| 列表卡 · 商品名称 | **固定 1 行** | 超出宽度尾端以「…」省略（如：骰宝(ID: 44)免费旋转/投…）|
| 列表卡 · 库存 / 积分 | 各 1 行 | 正常显示，不受名称长度影响 |
| 详情弹窗 · 商品名称 | **最多 1 行** | 超出以「…」省略 |
| 详情弹窗 · 商品简介 | 不限行数 | 过长时弹窗内可滚动；无简介则隐藏该区域 |

商品名称在列表卡**只占 1 行**，名称太长时尾端以「…」省略。请勿因名称过长撑高卡片或换行，以保持两列卡片高度一致、整齐对齐；完整名称于**商品详情弹窗**呈现。

> **QA 验证：** 上架一个超长名称商品，列表卡名称应压成 1 行 +「…」，且卡片高度与相邻卡片一致、不被撑高。

### 功能逻辑

- 库存为 0 时，商品卡显示「已搶光」遮罩（见第 7 页），且**商品卡不可点击**，无法进入商品详情。
- 商品下架时，商品卡不再展示于列表。

## 第 2 页　商品详情弹窗

**路径：** `商品列表 > 点击商品卡`  
弹窗含商品简介、「如何兑换 / 兑换条款」入口；积分充足可购买，不足时显示「余额不足」。

### 弹窗说明

点击商品卡弹出，沿用既有弹窗样式。本次新增：

- 标题栏显示分类名「**Share Treat**」（而非商品名）
- 商品名作为弹窗内容的副标题（左对齐，图片上方）
- 商品简介下方新增「**如何兑换 / 兑换条款 ›**」蓝字入口

### 弹窗字段来源

| 字段 | 来源 | 说明 |
|------|------|------|
| 商品名称 | 后台配置 | 弹窗副标题，显示于图片上方左对齐；如「McDonald's 中薯」 |
| 商品图片 | 后台上传 / 三方 | 置中正方形缩图 |
| 商品简介 | 后台配置 | 图片下方简短描述文字；无配置时隐藏该区域 |
| 如何兑换 / 兑换条款 | 三方 Brand API `redeemInfoUrl` | 蓝字入口，点击弹出兑换说明二级弹窗 |
| 购买价格 | 后台 | 「购买」按钮内显示折后积分 |

### 新增：如何兑换 / 兑换条款

| 项目 | 规格 |
|------|------|
| 位置 | 商品简介下方、底部按钮上方 |
| 样式 | 蓝色文字 + 右箭头「›」，可点击 |
| 点击行为 | 弹出「兑换说明」二级弹窗（覆盖在商品详情之上） |

### 底部按钮两种状态

| 状态 | 触发条件 | 按钮文案 | 点击行为 | 关闭弹窗 |
|------|----------|----------|----------|----------|
| 积分充足 | 会员积分 ≥ 商品折后价格 | 「购买 ⭐ XX」 | 跳转至**确认订单**（第 3 页） | 右上角「×」或点购买跳转 |
| 积分不足 | 会员积分 < 商品折后价格 | 「余额不足」 | 不可点击，无任何响应 | 仅右上角「×」可关闭 |

> **注意：**「余额不足」按钮外观与「购买」按钮一致（蓝底白字），仅文案不同且不可点击；会员须点右上角「×」关闭弹窗。

### 「兑换说明」弹窗规格

| 字段 | 来源 | 说明 |
|------|------|------|
| 弹窗标题 | 固定 | 「如何兑换 / 兑换条款」 |
| 正文内容 | 三方提供的 `redeemInfoUrl` | 以 WebView 内嵌展示三方链接页面，**内容由三方定义，我方不管控** |
| 关闭 | 固定 | 右上角「×」返回商品详情 |

> 条款内容含使用限制、不可退款等说明，是客诉高发点，须购买前可见。条款文字样式由三方页面定义，我方不做干预。

此弹窗以 **WebView** 内嵌方式加载三方提供的 `redeemInfoUrl`，页面内容由三方定义，我方仅负责加载展示，不做内容编辑。

### 功能逻辑

- 点「购买」进入**确认订单**（第 3 页）。
- 会员积分余额 < 商品折后价格时，「购买」按钮文案替换为「余额不足」且不可点，仅右上角「×」可关闭弹窗。
- 「兑换说明」弹窗内容拉取失败时，显示「加载失败，请稍后重试」+ 重试，不阻断商品浏览。

## 第 3 页　确认订单

**路径：** `购买 > 确认订单弹窗`

### 说明

沿用既有确认订单结构（商品价格 / 折扣 / 总价 / 返回 / 付款），**新增「兑换条款同意」勾选**，置于「付款」按钮上方。

### 新增：兑换条款同意勾选

| 项目 | 规格 |
|------|------|
| 文案 | 「我已阅读并同意《如何兑换 / 兑换条款》」 |
| 《如何兑换 / 兑换条款》 | 蓝字可点，弹出与第 2 页相同的「兑换说明」弹窗 |
| 默认状态 | **不勾选**（不可预勾，须会员主动勾选） |
| 付款按钮 | 未勾选时**置灰不可点**；勾选后恢复可点 |

### 功能逻辑

- 未勾选同意：付款置灰；点击置灰按钮可轻提示「请先阅读并同意如何兑换 / 兑换条款」。
- 「付款」前再次校验库存与积分余额；不足时提示并阻止下单。
- 付款 ＝ 扣减会员积分并向后台 / 三方发起兑换（派发）。成功后进入**购买成功**（第 4 页）。
- 建议服务端记录本次「同意条款」的时间戳与条款版本，便于日后查证。

> 扣分与三方派发的时序、失败回滚（退积分）以后台 / 服务端为准；前台只负责呈现结果状态。

## 第 5 页　兑换记录

**路径：** `积分商城 > 兑换记录`；或 `购买成功 > 查看订单`

### 改动点

1. **状态标签**：沿用系统既有三种状态（已完成 / 待审核 / 已拒绝）。自动审核 → 购买后直接显示「已完成」；人工审核 → 先显示「待审核」，审核通过后更新为「已完成」，审核拒绝则更新为「已拒绝」（系统全额退还本次扣减的积分）。
2. 其余结构（缩略图、时间、商品名、消耗积分、点击进详情）沿用既有逻辑。

### 字段规格

| 字段 | 说明 |
|------|------|
| 缩略图 | 商品图，缺图显示占位图 |
| 下单时间 | `YYYY-MM-DD HH:mm:ss` |
| 商品名称 | 如「McDonald's 中薯」 |
| 消耗积分 | 本笔扣减的积分 |
| 状态 | 待审核 / 已完成 / 已拒绝 |

### 功能逻辑

- 顶部下拉可按状态筛选（全部 / 待审核 / 已完成 / 已拒绝）。
- 三种状态（待审核 / 已完成 / 已拒绝）的记录均可点击，进入**订单详情**（第 6 页）。
- **已拒绝**：人工审核被拒的订单，列表显示灰色「已拒绝」标签；该笔扣减的积分已由系统全额退还至会员账户。
- 保留「数据同步存在延迟」提示文案。

> **前后台状态对照：** 前台「已完成」＝后台「审核通过」；前台「待审核」＝后台「待审核」；前台「已拒绝」＝后台「**审核未通过**」。三方派发、扣分与退分的时机以后台 / 服务端为准。

## 第 6 页　订单详情（既有页，新增「兑换信息」区块）

**路径：** `兑换记录 > 订单详情`　此页本已存在，本次仅在其内新增兑换信息。

### 改动说明

**订单详情为既有页面，本次不新增页面**，仅在原有「步骤条 + 订单信息」之下**新增「兑换信息」区块**。该区块展示第三方（BIZ TREATS）派发回传的数据，是会员实际「拿到货」的地方。

> **关键区分：「订单号」≠「兑换码」。** 「订单号」是识别这笔交易、用于查单 / 客服 / 对账；**会员拿去实体店核销的是「兑换码 / 兑换链接」，不是订单号**。两者在页面上分区块呈现，避免会员拿错。

### 订单信息区块（查单用 · 既有）

| 字段 | 来源 | 对应 API | 说明 |
|------|------|----------|------|
| 订单号（自家） | **自家系统** | = 传给三方的 `referenceNo` | 原「兑换单号」改称「订单号」，**与后台「商品兑换记录」列表的「订单号」用词一致**；接三方前后都由自家生成，建议直接作为 referenceNo 传给三方，保证 1:1 对应 |
| 商品名称 / 价格 | 自家 / 后台 | — | 沿用 |
| 三方单号（可选） | 三方 | `bizReferenceNo` | 默认**不在会员端显示**，放后台对账（后台已有「查看三方信息」）；仅当客服流程需会员报三方单号时，才以折叠 / 次要信息加入 |

> **单号命名约定：** 自家 ＝「订单号」（PR…，`referenceNo`，与后台列表一致）；三方 ＝「三方单号 / 供应商单号」（260…，`bizReferenceNo`，与后台用词一致）。避免两者都叫「单号」造成混淆。

### 兑换信息区块（换货用 · 新增）

以下字段**全部由三方派发后回传**，包含兑换码、兑换链接及有效期，内容以三方实际回传为准，我方不做预设或改写。

| 字段 | 类型 | 对应 API | 说明 |
|------|------|----------|------|
| 兑换码 | 文本 | `pin` | **码型：会员出示此码给店员核销**；与链接并存时，会员可同时看到兑换码与兑换链接 |
| 兑换链接 | 链接 + 打开 | `bizRedeemUrl` | **连结型：会员打开此链接兑换**，导向供应商兑换页 |
| 有效期至 | 文本 | `pinExpiredDate` | 三方派发时回传的到期日，**纯展示**，不由我方计算；时间格式化沿用既有逻辑；过期时码 / 链接置灰并提示「已过期，请联系客服」 |

### 按商品类型（pinType）显示规则

| 类型 | pinType | 显示内容 |
|------|---------|----------|
| PIN 码型 | `pin` | 仅显示**兑换码**（+ 有效期） |
| 连结型 | （url） | 仅显示**兑换链接**（+ 有效期），可「打开链接」 |
| 码＋链接型 | `urlWithPin` | **兑换链接 + 兑换码（两者皆有）** |

> 三方按商品决定回传 `pin` / `bizRedeemUrl` 二者其一或并存；前台按实际回传字段渲染，**不要写死**。无值字段不展示。

### 按状态显示规则

| 状态 | 触发条件 | 步骤条 | 兑换信息区块 |
|------|----------|--------|--------------|
| 待审核 | 后台设为人工审核，尚未派发 | 待审核（高亮）→ 已完成（灰） | 显示「**审核中，发放后可查看兑换信息**」，不显示码 / 链接 |
| 已拒绝 | 后台人工审核**拒绝** | 待审核 ✓ → **已拒绝（红色 ✓）** | 不显示兑换码 / 链接；**本次扣减积分已全额退还**，会员如有疑问透过「联系客服」 |
| 已完成 | 自动审核 / 人工审核通过后三方已派发 | 两步均 ✓ | 按 pinType 显示兑换码 / 兑换链接（或两者）+ 有效期 |

### 功能逻辑

- **如何兑换 / 兑换条款**：兑换信息区块下方提供蓝字入口，点击弹出与商品详情页相同的「兑换说明」弹窗（含可兑换门市清单）。会员拿到兑换码后仍可随时查阅去哪里核销。
- **不提供复制**：兑换码 / 兑换链接**不提供「复制」按钮**，会员直接读取使用（连结型仍可「打开链接」）。
- **打开链接**：连结型 / 码＋链接型提供「打开兑换链接」；码＋链接型同屏展示兑换码。
- **有效期**：纯展示三方回传的 `pinExpiredDate`，时间格式化沿用既有逻辑；已过期时码 / 链接置灰并提示「已过期，请联系客服」。
- **手动审核**：审核通过后才会有兑换数据；此前区块为「审核中」占位，审核通过后刷新自动展示（呼应列表「数据同步存在延迟」）。
- **审核拒绝（已拒绝）**：后台人工审核拒绝后，订单状态转为「已拒绝」，系统全额退还本次扣减的积分至会员账户；该状态下不产生兑换数据，故不显示兑换码 / 链接 / 有效期，也不显示「如何兑换 / 兑换条款」。
- **拒绝原因**：会员端**不显示拒绝原因**；已拒绝详情页下方常驻「对订单有疑问？联系客服」，会员如需了解透过客服查询即可（后台「备注」为内部字段、非必填，沿用既有逻辑）。
- 「联系客服」入口常驻。

## 第 7 页　商品列表 · 已搶光

**路径：** `积分商城 > Share Treat 分类`  
库存为 0 时，商品卡叠加半透明遮罩显示「已搶光」。

### 已搶光显示规则

| 项目 | 说明 |
|------|------|
| 触发条件 | 后台库存 = 0（或三方回传库存 = 0） |
| 遮罩样式 | 商品图片上叠加半透明黑色遮罩（opacity 约 45%），中央白色文字「已搶光」 |
| 积分颜色 | 积分数字正常显示，不变灰 |
| 库存文字 | 显示「库存 0」 |
| 点击商品卡 | **不可点击**，无跳转行为 |

### 功能逻辑

- 已搶光商品卡仍在列表中展示，不自动隐藏，排序由后台控制。
- 库存恢复 > 0 后，遮罩自动消失，恢复正常可购买状态。
- 遮罩为纯前端展示层，不影响商品数据结构。

> **QA 验证：** 将后台某商品库存设为 0，前台列表对应商品卡应显示「已搶光」遮罩，且商品卡点击无反应（不弹出商品详情）。

## 第 8 页　积分历史（现有功能，新增 Share Treat 记录条目）

**路径：** `积分商城 > 积分历史`  
Share Treat 购买会产生扣分与退分两种记录条目，按时间降序排列（最新在上）。

### 说明

积分历史为现有功能，本次不新增页面。Share Treat 购买流程需在积分历史产生以下两种条目。

### 页面结构

| 元素 | 说明 |
|------|------|
| 分类 Tab | 全部 / 收入 / 支出，沿用现有逻辑 |
| 数据范围提示 | 「*仅展示近30日数据」，沿用现有文案 |
| 排序 | 按时间降序，最新记录显示在最上方 |

### 两种条目规格

| 条目类型 | 触发时机 | 积分变动 | 系统文案 |
|----------|----------|----------|----------|
| 扣分 | 会员付款成功（含自动审核 & 人工审核商品） | 红色负数，如 **−16** | 「Redeem {商品名}」，如「Redeem McDonald's 中薯」 |
| 退分 | 人工审核**拒绝**后退还积分 | 绿色正数，与扣分金额相同，如 **+16** | 「Points refunded after product review rejection」 |

> **自动审核商品不产生退分记录：** 自动审核直接派发，无审核拒绝场景，积分历史只会有扣分记录。仅人工审核商品被拒时，才出现退分记录。

### 与兑换记录的对应关系

| 兑换记录状态 | 积分历史应有记录 |
|--------------|-----------------|
| 待审核 | 扣分记录（1 笔）；退分记录尚未产生 |
| 已完成 | 扣分记录（1 笔）；无退分记录 |
| 已拒绝 | 扣分记录（1 笔）＋退分记录（1 笔），两笔金额相同 |

### 功能逻辑

- 会员可在「积分历史」与「兑换记录」两个页面分别查到此次购买的相关记录，两者独立展示、互不替代。

> **QA 验证：** 触发人工审核拒绝后，① 兑换记录显示「已拒绝」；② 积分历史出现与扣分金额相同的绿色退分记录，时间晚于扣分记录；③ 会员账户积分已恢复。
